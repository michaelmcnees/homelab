# Tailscale Shared Portal Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a Tailscale-authenticated shared portal at `portal.mcnees.me` that shows each shared user only their allowed apps and reverse-proxies `/app/<name>` routes to in-cluster Arr services.

**Architecture:** Add a small Go `tsnet` application that becomes its own shareable Tailscale machine named `shared-portal`. The app uses Tailscale `WhoIs` on each incoming request for authentication, a mounted YAML config for per-user authorization, and Go's `httputil.ReverseProxy` for allowed app paths. Kubernetes deploys the app in the `auth` namespace with persistent tsnet state, an optional auth-key secret, a DNS-01 certificate, and no public ingress.

**Tech Stack:** Go 1.23, `tailscale.com/tsnet`, `gopkg.in/yaml.v3`, Kubernetes, Kustomize, cert-manager DNS-01.

## Global Constraints

- `portal.mcnees.me` is Tailscale-only.
- Public DNS may point at a Tailscale IP.
- Tailscale, not Pocket ID, authenticates shared users.
- Authorization is a declarative shared-user/app config.
- The portal should support multiple shared users from day one.
- The preferred user experience is a dashboard plus `/app/<name>` subpaths.
- Per-app hostnames or ports are compatibility fallbacks, not the target UX.
- Do not expose `portal.mcnees.me` through public Traefik or Cloudflare ingress.
- Do not require Pocket ID for shared Tailscale users.
- Do not invite shared users as normal tailnet users just to access these apps.
- Do not expose the Arr app services directly to Tailscale.
- Do not globally change an Arr app's URL base until the subpath compatibility spike proves it is safe.
